## Реклама YQL

Используя YQL, можно обрабатывать данные в системах [YDB](https://ydb.tech/docs/en/reference/?version=v25.3) и [YTsaurus](https://ytsaurus.tech/docs/en/yql/). Человек обычно разрабатывает запрос на диалекте SQL, обладащим полезными синтаксическими конструкциями вроде [именованных выражений](https://ytsaurus.tech/docs/en/yql/syntax/expressions#named-nodes), [анонимных функций](https://ytsaurus.tech/docs/en/yql/syntax/expressions#lambda), [именованных подзапросов](https://ytsaurus.tech/docs/en/yql/syntax/subquery) и другими мелочами вроде [FLATTEN BY](https://ytsaurus.tech/docs/en/yql/syntax/flatten).

Кроме синтаксических особенностей, у языка богатая система типов (есть структуры, варианты, перечисления, кортежи, списки, тэгированные типы), о которой можно узнать из [library/cpp/type_info](https://github.com/ytsaurus/ytsaurus/tree/9fd9d56162bb039ede5e1722ed2980a8bbcb92f9/library/cpp/type_info). Система типов дополняется множеством привычных функций: [ListMap](https://ytsaurus.tech/docs/en/yql/builtins/list#listmap), [ListFold](https://ytsaurus.tech/docs/en/yql/builtins/list#listfold), [DictLookup](https://ytsaurus.tech/docs/en/yql/builtins/dict#dictlookup), [Visit](https://ytsaurus.tech/docs/en/yql/builtins/basic#visit) (для `Variant`) и множества других. Функциональное программирование неплохо стыкуется с SQL, например, подзапрос возвращает список структур, а не таблицу, что позволяет совмещать оба типа стиля кодирования, переходя от одного в другой по необходимости, что должно быть удобно разработчикам ПО.

## Промежуточное представление

Уже сейчас появляется ощущение, будто можно отобразить на этот язык в какое-нибудь продвинутое лямбда-исчисление и писать запросы к СУБД на нормальном синтаксисе, похожим на Haskell, и больше никогда в жизни не трогать страшный SQL. Хотя транслировать такое в SQL было бы крайне неловко. В этот момент уже беспокоиться начинаешь: "Что же все наплодили своих SQL-диалектов и представлений плана запроса в чем-то на подобие реляционной алгебры, ведь кажется, что оно хорошо отображается на функциональное программирование, вот бы сделали что-то вроде LLVM, только для языков запросов".

В каком-то роде YQL как раз и реализует данную идею. На самом деле SQL -- это просто прикрытие, а YQL является функциональным DSL. SQL-синтаксис транслируется в промежуточное представление, модель YQL-запроса, представленное в программе в виде [TExprNode Graph](https://github.com/ytsaurus/ytsaurus/blob/9fd9d56162bb039ede5e1722ed2980a8bbcb92f9/yql/essentials/ast/yql_expr.h#L1790-L1799). Это не AST в привычном понимании, а более общая и менее удобная в использовании конструкция, представляемая текстом в виде YQLs синтаксиса ([s-expressions](https://en.wikipedia.org/wiki/S-expression)). То есть запросы к СУБД можно писать на чем-то очень похожим на [Clojure](https://clojure.org/) -- превосходно, и для ценителей локальный мем (пораженный Томас Шелби направляется в сторону указательного пальца): "Ты зачем аналитический запрос на s-expression-ах написал?!". Примеры YQLs можно посмотреть в [yql/essentials/tests/s-expressions/suites](https://github.com/ydb-platform/ydb/tree/cf1505412a89fe549f0d48aefe5cca6108a53b9d/yql/essentials/tests/s-expressions/suites), больше примеров будет далее при расммотрении YQStella.

Именно над `TExprNode Graph` производятся типизация, lowering, статические анализы, оптимизация, выполнение/планирование запроса и части именно этого представления транслируются в другой похожий функциональный язык, интерпретируемый или компилируемый в нативный код при помощи LLVM -- [MiniKQL](https://github.com/ytsaurus/ytsaurus/tree/9fd9d56162bb039ede5e1722ed2980a8bbcb92f9/yql/essentials/minikql). Такая декомпозиция позволяет поддерживать несколько синтаксисов на основе общей инфраструктуры.

## Первая программа

Вот пример простой программы на YQL. В качестве параметра запроса под именем `__x__` поступает число типа `Uint64`. К нему применяется функция `main`, по сути являющаяся id-лямбдой. Для ввода-вывода есть некоторая обвязка в виде `print`.

В языке есть ключевые слова:

- `(let name expr)` для связывания выражения с именем внутри блока.

- `(block '(stmt* (return expr)))` создает блок, внутри которого можно перечислять statements (let-связывания), он должен завершаться return-высказыванием.

- `(declare name type_expr)` объявляет параметр запроса.

- `(lambda '(params*) expr)` описывает анонимную функцию.

Есть куча "функций", называемых callables: `DataSink`, `ResFill!`, `Commit!`, `Apply`... Они мощнее функций, так как не имеют фиксированной сигнатуры и могут совершенно произвольным образом обрабатывать дочерние узлы. Это что-то вроде интринсиков. Подробнее о них позже.

Лямбды, кстати, тоже более мощные. Их тела не типизируются, а выполнить с ними можно только подстановку, как подстановку в лябда-исчислении. Что-то вроде макросов или шаблонов в C++.

```clojure
(
  (let print (lambda '(world rows) (block '(
    (let sink (DataSink 'result))
    (let options '('('type) '('autoref) '('unordered)))
    (let world (ResFill! world sink (Key) rows options 'pure))
    (return (Commit! world sink))))))

  (let main (lambda '(x) (Apply (lambda '(x) x) (== x (Uint64 '0)))))

  (declare __x__ (DataType 'Uint64))

  (let result (Apply main __x__))

  (let world (Apply print world (AsList (AsStruct '('result result)))))
  (return world)
)
```

Есть специальное значение `world` -- аналог [RealWorld](https://wiki.haskell.org/IO_inside) в Haskell -- для порядка вычислений с side-effects (callables с восклицательным знаком).

Рекурсивные связывания отсутствуют, встроенного fixpoint нет, лямбды раскрываются до исполнения запроса, потому при попытке сконструировать расходящееся выражение пользователь получит ошибку превышения лимита итераций из фрейворка для трансформаций, так что язык, кажется, не является Тьюринг-полным.

Кстати, запустить это чудо можно только собрав одну из утилит для разработчиков, например, [minirun](https://github.com/ydb-platform/ydb/tree/cf1505412a89fe549f0d48aefe5cca6108a53b9d/yql/essentials/tools/minirun), потому что как в [YT Query Tracker](https://ytsaurus.tech/docs/en/user-guide/query-tracker/about), так и в [KQP](https://github.com/ydb-platform/ydb/tree/cf1505412a89fe549f0d48aefe5cca6108a53b9d/ydb/core/kqp) возможности запустить YQLs нет -- очень жаль, держим локальный мем в голове.

```bash
minirun --program a.yqls --params-file p.yson --print-result
```

## SQL-транслятор

Все начинается с него. Данный компонент построен вокруг сгенерированного инструментом ANTLR4 дерева разбора. Ничего не предвещало беды, до даже тут есть интересности.

Во-первых, грамматика YQL (ссылка) перед генерацией кода подвергается препроцессингу. В YQL поддержаны 2 лексера для совместимости с ANSI SQL. Во-вторых, написан свой шаблон для генерации protobuf. Об этом можно узнать из статьи (ссылка). Эта штука убивает LSP и драматично увеличивает время компиляции.

На этапе SQL-трансляции еще есть информация о структуре запроса, что позволяет выдать относительно дружелюбную к пользователю диагностику, что и делается, значительно усложняя кодобазу.

Важно заметить, что транслятор порождает не текстовое представление YQLs, а именно программное, чтобы сохранить позиции узлов дерева в исходном файле.

Вокруг грамматики YQL, кроме угрюмого мудрейшего дуба-транслятора, развиваются жизнерадостные побеги молодых компонентов для инструментов разработки: модуль автодополнения (ссылка) и генератор конфигураций для подсветки синтаксиса в различные редакторах с единой точкой истины.

## Аннотация типов

TExprNode Graph не зависит от SQL-транслятора и может быть описан пользователем в текстовом представлении YQLs, потому не вправе рассчитывать на корректность программы. Более того, проблема обнаружения ошибок до запуска программы особенно остро стоит в контексте систем обработки больших данных, вычисления на которых могут длиться дни. Кроме того, информация о типах может быть использована при оптимизации, а также проверке корректности произведенных ими трансформаций. Само собой, программу мы запустить не можем, поэтому аппрокисимируем ее поведение типами.

В YQL реализована однонаправленная типизация в направлении inference. То есть типы синтезируются по графу снизу вверх.
